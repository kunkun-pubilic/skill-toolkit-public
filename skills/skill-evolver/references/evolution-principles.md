# 演化原则

## 目的

本文件定义增量升级技能时不可违背的原则。

## 不变量

1. 能力契约优先：
   - 技能定义的是触发条件下的行为，不是文字长度。
2. 以差分为中心：
   - 每次改动都要映射到能力差分和可感知收益。
3. 双模式先分类：
   - 先判断这次属于 `new-requirement` 还是 `conversation-harvest`，再决定工作流与证据形态。
4. 经验资产化：
   - 调用后的复盘收获只有在“跨未来任务可复用”时，才允许升级为参考资料或脚本。
5. 重排优先于尾部追加：
   - 当新能力影响既有逻辑时，必须重写受影响区块。
6. 真源纪律：
   - `SKILL.md` 必须声明资源使用与路由。
7. 先读后改：
   - 演化前先读取目标技能现有的 `SKILL.md`、相关 `references/`、相关 `scripts/`，避免重复发明已有资产。
8. 先验证再完成：
   - 没有确定性检查结果，不得声称完成。
9. 认知上游优先：
   - 若演化涉及方法论或判断标准的变更，先判断这个认知应住在知识库（Obsidian）还是技能本身。
   - 依赖链方向：知识库 → 技能 → 项目 → 启动层。认知变更从上游开始。
10. 双向路由：
    - `new-requirement`：检查目标技能的知识库依赖声明是否需要更新。
    - `conversation-harvest`：可复用的方法论级认知优先回灌知识库，技能只保留执行层收获。
    - 判断标准：若收获"换一个技能也成立"→ 知识库；若"只在这个技能的特定步骤才有意义"→ 技能。

## 渐进式披露规则

内容落位使用以下规则：

- `SKILL.md` 只保留：
  - 触发意图
  - 工作流决策点
  - 资源路由
- 以下内容放入参考资料：
  - 场景指导体量很大
  - 提示词密集
  - 来自复盘、且已抽象成可复用模式的提示词 / 注意点 / 检查清单
- 脚本只用于确定性检查或转换。
- 对话中的内容按以下规则分类：
  - 改变技能主行为或判断边界 → 回写 `SKILL.md`
  - 提示词模板、漏检点、变体化经验 → 落到 `references/`
  - 可重复、确定性、标准化步骤 → 落到 `scripts/`
  - 一次性上下文、客户专属噪音 → 只留在当次证据，不沉淀为资产
  - 方法论级认知变更（换一个技能也成立的判断标准）→ 标注回灌知识库建议，由用户确认后通过 obsidian-cli 执行

## 反模式信号

- 新功能只在文件末尾新增段落描述。
- 正文改了触发逻辑，但文件头元数据未同步。
- 新资源文件已加入，但 `SKILL.md` 未声明。
- 最终文件仍保留占位符文本。
- 把整段原始对话搬进参考资料，而不是提炼成复用规则。
- 每次复盘都新增文件，却不先检查是否应重写既有 参考资料/脚本。
- 方法论级认知被硬塞进单个技能的参考资料，而不是回写知识库让多个技能受益。
- 知识库笔记已更新但技能的知识库依赖表仍指向旧内容。
