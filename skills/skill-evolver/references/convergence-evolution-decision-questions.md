# 演进前决策问题库

本文件由收敛扫描拆出，承载 `SKILL.md` 中不应默认注入的长规则、模板或示例。`SKILL.md` 保留触发、流程、门禁和交接；执行到对应步骤时再读取本文件。

## 演进前必须回答的 5 个决策问题

每次启动前，按顺序回答这 5 个问题。答不清楚就先澄清，不要直接动手改。

**问题 1：来源是什么？**
- 新能力需求：用户明确要扩展能力。
- 小实验评估：修复后的小实验说明还需要新增能力。
- 任务复盘：一次调用结束后，要把可复用收获回灌。

不同来源的验证路径不同：新能力需求要对齐用户意图；小实验评估要保留 pressure scenario 和结论；任务复盘只提炼可复用部分，一次性上下文不进技能。

如果来源是“真实使用 / 小实验暴露流程缺口”，先记录产物类型和原生 proof surface：这次改的是 `skill`、`overlay`、`prompt-core`、`TASK-ledger`、`paper-experiment`、`frontend-design`、`deployment-tooling` 还是 `handoff-materials`；验证证据是否落在该类型的原生证据面上。验证面错配时，先路由到 `dev-essentials:verification-before-completion` 或对应下游补证，不把缺证据的经验直接沉淀为流程规则。

**问题 2：这个收获值不值得沉淀？**
用三个筛子过：
- **可复用**：下次遇到同类问题能不能直接调？
- **可复现**：别人传同样参数能得到同样判断吗？
- **可解释**：为什么这样判断，能写清楚吗？

三者缺一 → 留在对话证据或 knowledge/signal，不进技能。

**问题 2.5：沉淀成什么形态？**
三筛子只说明“值得记”，还要决定“记成什么”。沉淀任何流程性收获前先做一次“为什么链”：对着这次的具体步骤逐个问“这步为什么有效 / 换场景还成立吗 / 判断依据是什么”，追问到一个**场景无关**的判断标准为止。三种落点：
- **抽象得出可迁移内核** → 进 `SKILL.md` 正文当判断原则（默认目标），写成“先界定 X → 区分 Y → 判断标准是 Z”。
- **机械固定、必须照此顺序执行的步骤**（固定调用参数序、不可乱的发布命令序）→ 才进 `references/` 步骤清单。
- **抽象不出、只此一次成立** → 不进技能，留 run 当一次性案例。

反模式：把“这次怎么做”的完整动作录像整段搬进 `references/` 当复用模板——换场景即废。重过程是沉淀判断内核，不是沉淀步骤录像。

**问题 2.6：能否写成技能工程化设计卡？**
把本次收获压成一张卡，缺关键字段就先深访，不直接补提示词：

```text
技能工程化设计卡：
- 目标：本次演进要让下次同类任务更稳定地做对什么
- 输入：新增触发原话、必要材料、上游事实源
- 输出：新增或变化的交付物、路由结果、验证证据
- 组件：模型判断、确定性脚本、参考资料、资产、上下游 skill
- 组件关系：调用顺序、跳步条件、失败回环
- 状态载体：状态写进哪个文件、报告、manifest、TASK 或路线卡
- 验证面：机械验证、pressure scenario、人工验收各证明什么
- 删除条件：这次差分何时应该合并、降级或删除
```

卡片不是交付文档，而是落位前的判断工具。若组件关系或验证面说不清，只能输出 `needs-interview` 或 `pending-user-pilot`，不得声称演进闭环。

如果差分来自“一次完整任务过程记录 -> 技能改进建议”，先读 `references/task-process-improvement-proposal.md`。它只生成中文可审查建议，不自动修改权威插件源码，也不把维护机制并入日常任务门禁。

若复盘来自 Computer Use / 聊天取证 / 桌面巡检，按 `references/scenario-prompts.md` 的「Computer Use / 聊天取证复盘」段做采集期两项检查（分支是否走完 / 标签是否贴对）、规则资产化和分类输出。

**问题 3：这条变化住在哪一层？**
- 单技能内的提示词调整 → SKILL.md 或参考资料
- 确定性可重复的步骤 → 脚本
- 模板、图片、字体、样例数据、boilerplate、输出要复制的文件 → 资产
- 插件内普通资源默认仍住在子技能本地。即使多个技能复用结构约束、提示词模式、评分标准、示例库或一般脚本，也优先复制或保留在各自技能目录，方便后期独立分发
- MCP、hook、运行时配置按插件清单或对应配置维护，不当作技能资源
- 不因“多个技能都会用”创建插件级 `_shared/`、`shared/`、`common/`、`resources/` 或根部 `references/`。这类目录不是稳定的技能加载入口，容易让新对话漏读规则。
- **跨技能 / 跨宿主 / 跨仓库都成立的合并语义**（如 `contract-first`、`skip != pass`、AI 建议 边界）→ **路由到 `agent-prompt-core-maintainer`，回写到共享核心/剧本**
- 方法论级认知 → 用户自行回灌到笔记（技能不再代为引用）

升级判断错了，规则就会在错误的层被反复触发或错过。

**问题 4：重写还是新增？**
- 先读现有 `SKILL.md` / `references/` / `scripts/`
- 看既有区块能不能容纳这次差分
- **能容纳 → 重写区块**，避免只追加不整理
- 容不下 → 新增，但要说明为什么现有结构不够
- 若只是提示词密度变高，优先新建/改写参考资料；只有改变默认判断流程时才改 `SKILL.md`

**问题 4.5：六项注入是否完整？**
- 上下文：是否写清触发背景、路径、宿主、范围？
- 好示例：是否有 1-2 个短示例或参考资料？
- 输出约束：是否限定报告、文件、验证和禁止事项？
- 步骤：是否拆成可执行决策序列？
- 先思考：是否要求先诊断/计划再改？
- 角色：是否明确智能体是维护者、评审者、路由者还是打包者？

缺项处理顺序：先重写既有流程，再移到参考资料/脚本，最后才扩写 `SKILL.md`。不要为了满足六项检查把长示例、口号或场景全集塞进默认上下文。

**问题 4.6：资源压力是否触发拆分？**
- 预计新增后 `SKILL.md` 超 250 行 → 先移到 `references/`，再只在正文留读取条件。
- 已有或新增资产超过 3 个 → 先建/整理 `assets/` 或 `references/`，不要用正文描述替代文件。
- 出现反复执行的解析、校验、生成、同步动作 → 先建/改 `scripts/`。
- 本轮新增是参数化知识查表（评分标准、provider 矩阵、案例库、分支说明）→ 进 `references/`。
- 本轮新增是流程性收获 → 先按问题 2.5 抽象成正文内核；触发行数上限时正确动作是“抽象压缩进正文”，不是“整段外移”。流程录像不许当模板搬进 `references/`。
- 资源归属和"禁止插件根部共享池"按问题 3：插件内技能资源默认本地，多技能补同一原则就各自改，少量重复可接受。

**问题 5：本轮声明到哪一步？**
面对脆弱载体（OOXML / 公式 / 数据库 / 生产配置），声明边界必须精准：
- 诊断完成 ≠ 方案已验证 ≠ 生产已落地
- 只跑通单样本 → 写"诊断完成，待规模验证"
- 只验证了低侵入方案可行 → 写"方案已验证，待正式落地"
- 真正上线跑过 → 才能写"已生产落地"

越界声称是技能质量腐化的最大根源。
