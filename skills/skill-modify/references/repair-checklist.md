# 技能修复检查清单

## 触发与文件头元数据

- `SKILL.md` 具有合法 YAML 文件头元数据
- 文件头元数据仅包含 `name` 与 `description`
- `description` 明确说明：
  - 技能提供什么能力
  - 何时使用（具体触发场景）
- 保留至少 1 组“应触发/不应触发”样例，用于检查欠触发和误触发

## 引导质量

- 正文指令使用祈使式、面向执行
- 核心行为明确（范围、约束、输出契约）
- 可选章节中没有隐藏的关键规则
- 涉及 merge / review / 验收 时，主审查对象是否写清为 `contract + verification`，而不是默认读 diff

## 自由度校准

- 对脆弱操作提供明确约束步骤
- 对上下文敏感任务保留自适应推理空间
- 技能不强制不必要的固定流程
- 若技能涉及验证结论，是否区分流程状态与 `pass / fail / skip / uncertain`
- 失败后是否明确禁止通过删场景、放宽边界、改弱断言来自证通过

## 渐进披露

- 核心指导保留在 `SKILL.md`
- 大段细节放在 `references/`
- 脚本只用于确定性、可重复任务

## 资源对齐与清理（关键）

- `SKILL.md` 是资源定义真源（source of truth）
- `SKILL.md` 声明的 `scripts/`、`references/`、`assets/` 文件都存在
- `scripts/`、`references/`、`assets/` 中现有文件都在 `SKILL.md` 被声明（无孤儿资源）
- 从 `SKILL.md` 移除的资源，已同步删除文件与调用说明（如 `agents/openai.yaml`、其他文档/脚本中的路径）
- 目录内无悬空路径引用（引用存在但目标文件不存在）

## 噪音与漂移

- 无占位符文本（TODO/TBD/模板残留）
- 技能目录内无不必要的辅助文档
- 无无意义重复规则

## 三原则自检（最小化 / 重过程 / 关键词触发）

- **最小化**：单技能体量是否健康（≤ 250 行健康，超 350 必拆参考资料）？是否在做多职责（应拆成小技能让模型按需调用）？能写出一行 方法调用签名吗？
- **重过程**：正文是判断流程（写 how）还是抽象要求（写 what）？grep 命中"要认真 / 一步步思考 / 考虑风险 / 输出专业严谨"等限制护栏即标 P0
- **关键词触发**：描述字段是否显式列了用户原话 + 同义词（中英文）？纯"能力描述"靠模型语义推断会漏触发或幻觉误触发

## 验证

- `quick_validate.py` 通过
- 技能本地审计通过
- `package_skill.py` 可成功产出 `.skill`（交付前）
- 对本次删除项执行 `rg -n "<removed-name>" <skill_dir>`，确认无残留引用
- 改动有命令证据支撑
- 若涉及 合并语义，至少保留 1 组”改前默认读 diff / 改后默认看 contract summary”的对照证据

## 修复启发式规则

- 文件头元数据保持聚焦：`SKILL.md` 中仅保留 `name` 和 `description`。
- 触发逻辑写在 `description`，不要藏在正文章节里。
- 触发优化要防过拟合：优先扩展”意图类别”，不要只追加具体句子清单。
- 正文保持祈使式、可执行。
- `SKILL.md` 是资源定义真源，不允许旁路声明。
- 优先使用示例与简洁规则，避免冗长叙述。
- `SKILL.md` 保持紧凑，变体细节放到 `references/`。
- 仅在需要确定性、可重复检查或转换时添加脚本，脚本不得承载价值判断。
- 从 `SKILL.md` 移除的资源，不得以”备份”名义继续留在目录中。
- 不在技能内创建额外文档，如 `README.md`、`CHANGELOG.md`、安装指南等。
