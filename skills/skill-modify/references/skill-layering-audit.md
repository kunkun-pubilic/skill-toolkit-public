# Skill 分层审计

用于判断一个 skill 是否把事情放在了正确层级。它不替代 `quick_validate.py`，而是给修技能时的人和 Agent 一个共同口径：哪些交给 LLM，哪些交给代码，哪些只做按需参考。

修 skill 时同时按 `agent_prompt_core/shared/core/上下文信息分层_规则.md` 检查信息归属：`SKILL.md` 不应承载项目状态、长期方法论全文或机械流程。项目状态回到项目 knowledge/run，长期方法论下沉到 reference/playbook/core 候选，可机械执行的流程优先进入 scripts/gate。

## 先分固定与动态

审计时先回答一个问题：这一步是否需要当场理解用户意图、项目上下文、风险取舍或模糊证据？

- 不需要，且同样输入应得到同样输出：进入固定层，优先脚本化、模板化或写成明确 workflow。
- 需要，且结论依赖上下文或价值判断：进入动态层，保留给 LLM，用提示词写清判断标准和停下条件。
- 只是长规则、案例、rubric 或 provider 矩阵：进入按需参考层，不放进默认正文。
- 只是复制用材料、模板、图片、fixture 或 boilerplate：进入资源层。

固定层不是“更重要”，动态层也不是“更高级”。分层目标是让模型只处理脚本工作流无法稳定完成的部分。

## 组件关系审计

分层之外还要看它在 harness 图里的关系。目标不是画完整大图，而是找出黑盒节点、重复节点和无验证节点。

每次修技能至少回答：

- 目标：这个 skill 当前承担的稳定判断流程是什么。
- 输入：它实际吃哪些触发词、材料、上游事实源和用户裁决。
- 输出：它实际产出什么文件、路由结果、验证证据或交接包。
- 边界：哪些相邻意图不归它，分别归哪个 skill/plugin/playbook。
- 依赖：它依赖哪些 references、scripts、assets、上游 skill 或运行时工具。
- 验证面：它的结果如何证明有效，是否只有自然语言自报。
- 删除条件：什么证据出现时应合并、下沉、脚本化、禁用或删除。

组件关系结论只使用这些动作词：`keep / merge / downshift / script / delete-candidate / observe / needs-user`。`delete-candidate` 只是候选；删除技能、运行时配置或长期资产仍需要用户确认。

## 分层原则

| 层 | 对应载体 | 应该放什么 | 不应该放什么 |
|---|---|---|---|
| 动态判断层 | `SKILL.md` | 触发边界、负面边界、判断流程、何时问用户、何时停止、何时调用脚本或读取参考 | 长示例、provider 全表、机械解析、重复模板、完整评分明细 |
| 固定执行层 | `scripts/` 或明确 workflow | 同输入应同输出的解析、校验、转换、同步、打包、评分、manifest 生成 | 价值判断、用户意图取舍、模糊场景裁决 |
| 按需参考层 | `references/` | 长规则、schema、示例库、评分 rubric、provider 矩阵、故障说明 | 当前轮每次都必须读的主流程 |
| 资源层 | `assets/` | 模板、fixture、图片、字体、boilerplate、最终输出资源 | 需要模型理解的正文说明 |

## 审计问题

对目标 skill 逐项标注：

1. 哪些步骤需要实时理解上下文或价值判断？这些保留在 `SKILL.md`。
2. 哪些步骤 100 次执行都应该一致？这些应下沉到 `scripts/`。
3. 哪些内容只有特定分支才读？这些应移动到 `references/`。
4. 哪些资源只是输出材料？这些应放入 `assets/`。
5. `SKILL.md` 是否仍能写出清楚的方法调用签名？
6. `description` 是否只负责触发，不承载流程正文？
7. 是否存在插件根部共享资料导致子技能无法独立分发？
8. 是否有脚本输出不可纠错，只告诉 Agent pass/fail 而不给 repair route？
9. 固定层是否已经有明确命令、脚本入口、schema 或 workflow？如果没有，是否只是被长提示词伪装成流程？
10. 动态层是否写清了判断标准、反例、停下条件和用户确认门？如果没有，脚本化会把错误判断固化。

## 脚本化门槛

满足以下任意 3 条，优先考虑进入 `scripts/` 或明确 workflow：

- 输入输出结构稳定。
- 失败原因可以机械枚举。
- 执行结果可以用 exit code、JSON gate 或文件存在性验证。
- 同类任务会反复执行。
- 人工判断只发生在脚本运行前或运行后，不发生在脚本内部。

以下情况不脚本化，保留给 LLM：

- 用户真实意图不明确，需要先问问题。
- 需要判断“做不做”“先做哪一层”“是否越界”。
- 需要比较风险、成本、证据强弱。
- 需要把文章、复盘或项目语境转成系统改造建议。

## 验收

- `quick_validate.py --json <skill-dir>` 通过。
- 修改触发、流程或输出行为时，至少有一个 pressure scenario；没有真实新对话证据时状态写 `pending-user-pilot`。
- 新增或保留的脚本失败时能给出修复方向，而不是只输出原始异常。
- 用户侧说明先用自然中文解释结论，机器字段放在后面。
