# 技能修复 pressure scenario 闭环

用于修复已有技能后验证“用户可感知行为是否真的改善”。确定性校验只能证明文件合法，不能证明新提示词满足用户要求。

pressure scenario 必须复现旧问题或相邻技能冲突，让修复后的技能必须表现出可观察差异。不要用只会检查格式的 toy prompt 当验收。

## 必做条件

以下情况至少产出 1 个 pressure scenario：

- 修改了 `description`、触发关键词或负面边界。
- 改了默认流程、输出契约、验证门或资源路由。
- 修复目标来自用户反馈、误触发、漏触发或旧输出失败。

如果当前会话无法可靠加载新版技能，输出 `pending-user-pilot` 和新对话提示词。真实新对话触发、runtime 可见性和分发验证交给 TASK-007；当前任务不能用本线程模拟结果冒充通过。

## 最小小实验简述

```text
pressure scenario 简述

待测技能：<技能或 plugin:skill>
变更类型：修改
本轮修复要解决什么：<一句话>
旧版失败点或旧输出：<摘要>
新版应该改变的行为：<一句话>
测试提示词：<复现旧问题的真实用户风格小任务；包含路径、约束或相邻技能干扰>
预期差异：
- <新版必须避免旧失败点>
评判方式：
- 通过：<旧问题消失且没有明显新增误伤>
- 部分通过：<旧问题部分改善但仍有缺口>
- 失败：<旧问题仍存在或出现更严重误触发>
当前状态：通过 / 部分通过 / 失败 / pending-user-pilot
TASK-007 输入：<若 pending-user-pilot，把本块原样交给 TASK-007>
```

## 新对话提示词

```text
你是独立技能 pressure scenario 评审者，不参与刚才的修复。请在真实新对话中判断新版技能是否自然触发，并且是否真的修复了旧问题，而不是只改了文档。

待测技能：
<技能名称和路径；插件内技能写 plugin:skill>

旧版问题：
<1-3 条旧问题或失败输出摘要>

新版修复目标：
<1-3 条必须观察到的行为变化>

请执行这个小任务：
<能复现旧问题的真实用户风格提示词>

评判方式：
- 通过：旧问题被修复，并给出证据。
- 部分通过：有改善，但仍有未闭环点。
- 失败：旧问题仍存在、触发更差，或输出不符合要求。

输出格式：
1. 评估结论：通过 / 部分通过 / 失败
2. 证据：<具体输出/行为证据>
3. 缺口：<仍缺什么>
4. 给技能维护者的反馈：<下一轮应改哪里>
```

## 回灌

- 通过：把小实验作为完成证据。
- 部分通过 / 失败：先提炼失败模式，再决定走 `skill-modify` 继续修，或走 `skill-evolver` 增量演化。
- 如果失败暴露的是全局规则问题，升级到 共享核心/剧本，不塞进当前技能。
- `pending-user-pilot`：不声称行为闭环；把 pressure scenario 交给 TASK-007 做真实新对话测试。
