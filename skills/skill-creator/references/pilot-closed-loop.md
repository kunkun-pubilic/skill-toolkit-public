# 新技能 pressure scenario 闭环

用于验证新建 Skill 是否真的改善用户可感知行为。结构校验只能证明文件合法，不能证明新 Skill 会在真实任务中正确触发和执行。

## 最小小实验

```text
pressure scenario 简述

待测技能：<技能名称和路径>
创建目标：<这个技能要稳定解决什么判断流程>
相邻误触发风险：<容易被哪个 skill/plugin 抢走，或容易抢哪个任务>
测试提示词：<真实用户风格的一句话或小任务>
预期行为：
- <新版技能必须自然触发>
- <新版技能必须读取/执行的资源或脚本>
- <新版技能必须避免的相邻误触发>
评判方式：
- 通过：触发、输出和验证面都符合创建目标。
- 部分通过：结构可用，但触发或验证仍有缺口。
- 失败：没有触发、误触发，或输出绕过关键门禁。
当前状态：通过 / 部分通过 / 失败 / pending-user-pilot
```

## 规则

- 没有真实新对话证据时，状态只能写 `pending-user-pilot`。
- 如果小实验失败，先走 `skill-modify` 修触发/结构；需要新增复用能力时再走 `skill-evolver`。
- 如果失败原因是全局路由或宿主兼容，更新 registry / compatibility matrix，不把问题硬塞进当前 Skill 正文。
