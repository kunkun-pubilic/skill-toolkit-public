# Skill 设计笔记

Skill 的重点不是把提示词存起来，而是把一套会反复发生的工作接口稳定下来。

## 一个好 Skill 至少包含四层

### 1. 触发边界

`description` 应该告诉 Agent 什么时候用这个 Skill，也要说明什么时候不要用。触发边界不清，Skill 多了以后就会互相抢任务。

### 2. 判断流程

正文保留最关键的判断顺序。不要把所有资料塞进正文；正文越长，越容易变成另一个难维护的提示词。

### 3. 参考资料

长资料放到 `references/`。Agent 需要时再读，既省上下文，也方便更新。

### 4. 机械动作

能机械验证的部分放到 `scripts/`、工具或检查器里。不要让模型凭感觉判断文件是否齐、字段是否对、链接是否存在。

## 什么时候不要写 Skill

- 只发生一次的任务；
- 纯背景信息；
- 没有稳定步骤的临时沟通；
- 已经有成熟工具可以直接解决的事情；
- 需要外部授权、支付、发布、删除但没有清楚停手机制的流程。

## 从小场景开始

不要一上来做「全能工作流」。先找一个经常重复的小动作，例如：

- 每次视频发布前整理标题、简介、标签；
- 每次写 README 前检查是否泄露内部路径；
- 每次交付前确认文件、尺寸、hash、链接是否对齐；
- 每次调研后把来源、结论、限制分开。

小 Skill 先跑通，再考虑是否需要 Plugin。
